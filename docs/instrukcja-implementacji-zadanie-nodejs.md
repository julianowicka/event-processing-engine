Temat: Event Processing Engine Opis biznesowy Firma posiada wiele zewnętrznych integracji wysyłających różnego rodzaju zdarzenia (events). Dla przykładu załóżmy, że każde zdarzenie: • przychodzi asynchronicznie, • może pojawić się wielokrotnie, • może przyjść w złej kolejności, • może zostać opóźnione, • może częściowo nadpisywać poprzedni stan. Twoim zadaniem jest stworzenie silnika przetwarzania zdarzeń dla systemu zamówień. Dane wejściowe System otrzymuje eventy w formacie JSON: JSON { } "eventId": "evt-1001", "orderId": "ord-501", "type": "ORDER_UPDATED", "timestamp": 1710001000, "payload": { "status": "PAID", "amount": 199.99 } Możliwe typy eventów • ORDER_CREATED • ORDER_UPDATED • PAYMENT_CAPTURED • ORDER_CANCELLED • REFUND_ISSUED Problem Eventy: • mogą przyjść wiele razy, • mogą przyjść w innej kolejności niż zostały wygenerowane, • mogą mieć konfliktujące dane, • nie wszystkie eventy są poprawne, • część eventów może być przestarzała. System ma utrzymywać: 1. Aktualny stan zamówienia. 2. Historię zmian. 3. Audit log decyzji silnika. Wymagania 1. API Stwórz REST API: POST /events Przyjmuje batch (tablicę) eventów. GET /orders/:id Zwraca: • aktualny stan zamówienia, • historię zmian, • informację które eventy zostały odrzucone i dlaczego. GET /stats Zwraca: • liczbę poprawnych eventów, • odrzuconych eventów, • duplikatów, • średni czas przetwarzania. Reguły biznesowe Deduplication Jeśli eventId już istnieje: • event ma zostać zignorowany, • ale zapisany w audit log. Ordering Jeśli event przyjdzie starszy niż ostatni zaakceptowany event: • system musi zdecydować:  o czy go odrzucić, o czy częściowo zastosować, o czy zmergować. Kandydat musi sam zaproponować strategię. State transitions Niektóre przejścia są niedozwolone. Przykład: • CANCELLED → PAID powinno być niemożliwe. Ale: • PAID → REFUNDED jest poprawne. Kandydat sam definiuje sensowny state machine. Partial updates Niektóre eventy zawierają tylko część danych. Przykład: JSON { "payload": { } "amount": 250 } Silnik nie może usuwać brakujących pól.
