# Processing Flow

This document describes ingestion and asynchronous event processing.

## Ingestion Flow

1. Validate that the request body is a JSON array.
2. Insert each item into `raw_incoming_events` in the same order as the request array.
3. Create one `PENDING` `event_processing_jobs` row for each raw delivery.
4. Commit the ingestion transaction.
5. Return `QUEUED` results with `incomingEventId` and `jobId`.

`POST /events` does not validate business rules, deduplicate events, update
orders, write history, or update processing stats.

## Worker Claim Query

The worker processes jobs in raw delivery order:

```sql
SELECT epj.*
FROM event_processing_jobs epj
JOIN raw_incoming_events rie ON rie.id = epj.raw_incoming_event_id
WHERE epj.status = 'PENDING'
  AND epj.available_at <= CURRENT_TIMESTAMP
ORDER BY rie.id ASC
LIMIT 100;
```

Ordering is based on `raw_incoming_events.id ASC`. `event_timestamp` represents when
the event was generated by the external system and is used only for merge rules.

## Job Processing Flow

Each job is processed in its own transaction.

1. Mark the job as `PROCESSING`.
2. Start the processing timer.
3. Load the related `raw_incoming_events` row.
4. Extract minimal identifiers from the raw delivery.
5. If `eventId` is missing, write a `REJECTED` decision and mark the job `DONE`.
6. Try to insert `eventId` into `processed_event_keys`.
7. If the key already exists, write a `DUPLICATE` decision and mark the job `DONE`.
8. Validate the full event shape and payload.
9. If structurally invalid, write a `REJECTED` decision and mark the job `DONE`.
10. Load current order state and field versions.
11. Apply state machine rules.
12. Apply field-level merge rules.
13. Update `orders`.
14. Update `order_field_versions`.
15. Insert `order_history` when state changed.
16. Insert `event_decisions`.
17. Update aggregate stats.
18. Mark the job `DONE` or `FAILED`.
19. Commit the job transaction.

## Batch Result

The response contains one queued result per input item. The order of response
results matches the order of input events.

## Determinism

Raw deliveries are processed by `raw_incoming_events.id ASC`. Out-of-order event
generation is handled by timestamps and field versions during merge.
